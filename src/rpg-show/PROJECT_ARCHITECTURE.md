# Архитектура приложения для НРИ-шоу с голосованием зрителей

## 1. Цель проекта
Создать сценическое веб-приложение для интерактивного НРИ-шоу, где:
- зрители по QR-коду открывают пульт и голосуют;
- мастер запускает опросы в отдельной админ-вьюшке;
- на сценическом экране показываются карточки персонажей, таймер опроса и результаты.

## 2. Роли и интерфейсы

### 2.1 Зритель (`/rpg-show/audience`)
- Подключается по QR-коду (короткая ссылка + `showId`).
- Видит текущий активный опрос и кнопки `1`, `2`, `3`.
- Может отправить один голос за опрос (с возможностью изменить до завершения, если это разрешено настройкой шоу).
- Получает статус: «голос принят», «опрос завершён», «ожидание следующего опроса».

### 2.2 Мастер (`/rpg-show/master`)
- Запускает и останавливает опрос.
- Видит таймер и статистику в реальном времени.
- Переключает сценический экран между состояниями: `IDLE`, `POLL`, `RESULT`.

### 2.3 Экран сцены (`/rpg-show/screen`)
- Показывает статичный слой: карточки персонажей.
- Показывает динамический слой:
  - во время опроса: таймер;
  - после завершения: результаты и статистику (кол-во/проценты).

## 3. Доменная модель

### 3.1 Сущности
- `Show`
  - `id`, `name`, `status`, `createdAt`
- `Poll`
  - `id`, `showId`, `status`, `startedAt`, `endedAt`
- `Vote`
  - `id`, `pollId`, `voterKey`, `optionIndex`, `createdAt`

### 3.2 Идентификация зрителя
- Без логина: выдаётся `voterKey` (UUID в cookie/localStorage).
- Для анти-спама:
  - ограничение 1 активного голоса на `pollId + voterKey`;
  - rate limit на IP;

## 4. Жизненный цикл опроса
1. Мастер нажимает «Старт» -> создается опрос со статусом `ACTIVE`, запускается серверный таймер.
2. Зрители отправляют/обновляют голос до истечения времени.
3. По таймеру (или вручную) -> статус `FINISHED`.
4. Backend фиксирует итог, пушит результаты в `screen` и `master`.

Важно: завершение опроса должно происходить на сервере, а не в клиенте.

## 5. Realtime-события (WebSocket)

### 5.1 С сервера к клиентам
- `show.state.updated` — текущее состояние шоу.
- `poll.started` — старт активного опроса.
- `poll.timer.tick` — обновление оставшегося времени.
- `poll.stats.updated` — промежуточная статистика.
- `poll.finished` — финальные результаты.

### 5.2 От клиентов на сервер
- `audience.vote.submit` `{ pollId, optionIndex }`
- `master.poll.start` `{ pollId }`
- `master.poll.stop` `{ pollId }`
- `master.screen.mode` `{ mode: IDLE|POLL|RESULT }`

## 6. REST API (минимум)
- `POST /api/shows` — создать шоу.
- `GET /api/shows/:showId/state` — текущее состояние.
- `POST /api/shows/:showId/polls` — создать опрос.
- `POST /api/polls/:pollId/start` — запуск.
- `POST /api/polls/:pollId/stop` — досрочная остановка.
- `GET /api/polls/:pollId/results` — результат.

## 7. Экранные состояния (FSM)
- `IDLE` — только карточки персонажей.
- `POLL_ACTIVE` — вопрос + варианты + таймер.
- `POLL_RESULT` — итоги опроса.

Переходы инициируются мастером и/или серверным жизненным циклом опроса.

## 8. Нефункциональные требования
- Отказоустойчивость
  - при перезагрузке клиента происходит ре-синхронизация состояния;
  - экран при дисконнекте показывает «ожидание соединения» и последний стабильный кадр.
- Нагрузка MVP: 300-1000 одновременных зрителей.
- Безопасность:
  - мастерская вьюшка доступна конкретному списку ползователей, авторизованных через google
  - CORS и rate limit;
  - валидация payload для всех WS/REST запросов.
